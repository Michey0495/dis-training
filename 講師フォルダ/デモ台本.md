## Session 02: コンテキスト設計と仕様駆動開発（100分）

### 座学パート: コンテキストエンジニアリング（20分）

スライド/HTML教材ベースで進める。

伝えたいことは3つ:

1. Copilotの出力品質は渡す文脈で決まる
   - 何を開いているか、何を指示するか、Agent MDがあるか。この3つの設計が全て
   - プロンプトの小手先テクニックではなく「AIに渡す情報の設計」

2. Claude Codeから降りてきた機能群
   - copilot-instructions.md は Claude Code の CLAUDE.md が元ネタ
   - エージェントモードも同じ系譜
   - 「新機能」ではなく「AI開発ツールの設計思想」として理解する

3. コンテキスト設計はチーム施策
   - 個人がプロンプトを工夫するのは属人的
   - Agent MDをリポジトリに入れるのはチーム施策
   - 全員のAI出力品質が揃う

### TRY IT: 文脈なし/あり比較（15分）

話すこと:
「ここから手を動かします。まずは文脈を渡さずにCopilotに指示してみます。」

操作: HTML教材の TRY IT セクションに沿って進める

Step 1: 文脈なしで指示
```
議事メモからユーザーストーリーを抽出してください
```

話すこと:
「何が起きましたか。Copilotは『議事メモ』が何かわからないので、架空のユーザーストーリーを出してきます。これが文脈なしの状態です。」

Step 2: requirements.md を作成して #file: で渡す

```
#file:docs/requirements.md

この議事メモからユーザーストーリーを抽出してください。
メモに書かれていない要件は追加しないでください。
```

話すこと:
「同じ指示に #file: を足しただけです。出力が全く変わったのがわかると思います。プロンプトの一文字も変えていない。変えたのは渡した文脈だけです。」

### TRY IT: 制約指示の体験（10分）

話すこと:
「次は制約指示の効果を試します。」

操作: HTML教材に沿って制約なし/ありの比較

制約なし:
```
#file:docs/requirements.md
このメモからPythonのデータ分析スクリプトの仕様書を作成して
```

制約あり:
```
#file:docs/requirements.md
このメモからPythonのデータ分析スクリプトの仕様書を作成してください。
以下の制約を守ってください:
- メモに記載のない機能は追加しない
- 認証・ログイン機能は含めない
- 出力形式はMarkdownテーブル
```

話すこと:
「制約を入れないと、AIは親切心で勝手に要件を足します。『認証が必要では？』『ログ出力もつけましょう』。メモのどこにもそんな話は出ていない。制約指示は防御壁です。」

### TRY IT: コメントドリブン開発（10分）

操作: practice.py を作成してコメントを先に書く

```python
# CSVファイルを読み込む

# 部門ごとにグループ化して売上合計を計算する

# 結果をDataFrameで返す
```

話すこと:
「コメントだけ書いてカーソルを置くと、Copilotが本体を補完します。Tabで受け入れてください。」

操作: 受講者が各自で試すのを見守る。Tab補完が出ない場合は Enter で改行してから少し待つよう伝える

### デモ: 仕様書の構造化（15分）

操作:

```
#file:docs/requirements.md

上記の要件をもとに、以下の構成でMarkdown仕様書を作成してください。
- 機能一覧（テーブル形式）
- データ構造（入力CSVのカラム定義）
- 出力仕様（出力ファイル名、カラム構成）
- 処理フロー（箇条書き）
```

話すこと:
「仕様書の構成を指定しています。テーブル形式にする、カラム定義を含める、処理フローを箇条書きにする。この構成指示がないと、AIはだらだらとした文章を返します。構造化の指示を与えるかどうかで出力品質が決まります。」

操作: 出力をコピーし spec.md として保存

```bash
git add spec.md
git commit -m "仕様書を作成"
```

### ハンズオン: 各自で仕様書を作成（30分）

話すこと:
「ここからは各自で作業してください。議事メモから要件を抽出し、制約指示を付けて、spec.mdとして仕様書を構造化してください。最後に git commit で保存するところまでが目標です。」

巡回時の確認ポイント:
- #file: でメモを参照できているか
- 制約指示が入っているか
- spec.md をファイルとして保存しているか（チャット上だけで終わっていないか）
- git commit まで到達しているか

## Session 03: Agent MD × GitHub Copilot制御（100分）

### 座学パート: Agent MDの設計（20分）

画面: templates/copilot-instructions.md を開く

話すこと:
「.github/copilot-instructions.md というファイルをリポジトリに置くと、そのリポジトリでGitHub Copilotを使う全員がこのルールに従った出力を得ます。個人設定ではなくチーム設定です。」

「書くべきは4カテゴリ。コーディング規約、推奨パターン、禁止事項、出力フォーマット。特に禁止事項が効きます。bare except禁止、print()デバッグ禁止と書くだけでCopilotがそれを守ります。」

HTML教材の講師開示「.github フォルダの全体構成」を開いて見せる:
- copilot-instructions.md（常時適用）
- instructions/（ファイル種別ごと）
- AGENTS.md（Agent Modeの行動制約）
- skills/（再利用タスク定義）
- .prompts/（プロンプト集）

話すこと:
「今日の研修ではcopilot-instructions.mdを中心に扱いますが、配布素材のZIPにはここに見えている全ファイルが入っています。研修後に展開して確認してみてください。」

### ハーネスエンジニアリングの紹介（5分）

HTML教材のTopic 02「ハーネスエンジニアリングという考え方」を見せながら進行。

話すこと:
「ここで1つ大事な概念を紹介します。2026年のAI開発で最も注目されているのが『ハーネスエンジニアリング』です。ハーネスとは馬を制御する手綱のこと。AIエージェントは馬、copilot-instructions.mdは手綱です。」

「Agent = Model + Harness。モデルの性能差ではなく、モデルの外側にある設定・制約・フィードバックループの設計が出力品質を決める。この考え方はMartin Fowler氏もOpenAIも公式に取り上げています。」

「今日学んでいるcopilot-instructions.mdの書き方は、Claude CodeでもCursorでもCodexでも同じ思想で使えます。ツールが変わってもハーネスの設計力は持ち越せる。」

講師開示「ハーネスを構成するファイル群の全体像」を開いて見せる（時間があれば）。

### デモ: Agent MD 適用前後の比較（15分）

ここが研修で最もインパクトがあるパート。必ず見せる。

操作1: Agent MDを一時的にリネーム

```bash
mv .github/copilot-instructions.md .github/copilot-instructions.md.bak
```

操作2: Copilot Chat で入力

```
CSVファイルを読み込んで、部門ごとの売上合計を集計する関数を書いて
```

話すこと:
「Agent MDがない状態です。出力を見てください。」

指摘ポイント（出力によるが典型的に起こること）:
- 型ヒントがない
- docstringがないか、スタイルがバラバラ
- 変数名が日本語混じり
- bare exceptが使われている可能性

操作3: Agent MDを復元

```bash
mv .github/copilot-instructions.md.bak .github/copilot-instructions.md
```

操作4: 新しいチャットで同じ指示を再入力

```
CSVファイルを読み込んで、部門ごとの売上合計を集計する関数を書いて
```

話すこと:
「同じ指示です。Agent MDがある状態。出力の差を見てください。型ヒントが付いている。docstringがGoogle Style。変数名が英語。これが『設定して使う』と『設定しないで使う』の差です。」
